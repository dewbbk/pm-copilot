# Контейнер тасок эпика

> **Цель**: Превратить эпик из декомпозиции цели в конкретный набор юзер стори / тасок, из которых PM выбирает первую для проработки.
> **Формат**: Четыре фазы последовательно — выбор → сбор → сохранение → выбор таски. Не перепрыгивай фазы.
> **Цикл**: После завершения таски (PRD утверждён) → возврат в epic для следующей таски того же эпика.

## Фаза 1: Выбор эпика

Если PM входит в epic впервые (нет active_epic_id):

**Вопрос 1 — Выбор эпика:**
«У вас определены эпики для цели "[goal.text]». С какого начнём?»

Покажи список эпиков из `goals[active].epics[]`:
```
Эпики цели "[goal.text]":
1. [epic.title] — [epic.status]
2. [epic.title] — [epic.status]
...

Какой берём первым? (номер)
```

Рекомендация: предложить эпик со status: planned, который имеет наивысший приоритет (по roadmap из pm-copilot-goal).

Если PM уже работает с эпиком (active_epic_id заполнен) — перейти к Фазе 2.

## Фаза 2: Сбор юзер стори / тасок

**Вопрос 2 — Способ сбора:**
«Как собираем юзер стори для эпика "[epic.title]"?»

Варианты:
- **[1] Генерируем вместе** — pm-copilot генерирует варианты через generation-инструмент, PM выбирает и дополняет
- **[2] Комбо** — PM называет то, что знает/уже есть, pm-copilot генерирует недостающее



### Режим [1]: Генерируем вместе

1. Вызвать generation-инструмент: «Сгенерируй 5-7 юзер стори для эпика "[epic.title]" в рамках цели "[goal.text]"»
2. Показать список PM
3. PM выбирает подходящие и добавляет свои
4. Перейти к Фазе 3

### Режим [2]: Комбо

1. Спросить: «Какие таски ты уже знаешь, что нужны?»
2. Записать
3. Спросить: «Теперь давай генерирую дополнения?»
4. Если да — вызвать generation-инструмент с учётом уже названных
5. Показать расширенный список
6. Перейти к Фазе 3


## Фаза 3: Сохранение и приоритизация

Сохранить все собранные таски в `goals[active].epics[active].tasks[]` ProductState:

```yaml
tasks:
  - id: task-YYYY-MM-DD-HHMM-NN
    title: "[название таски]"
    status: draft
    prd_id: null
```

Показать PM итоговый список:
```
Эпик: [epic.title]
Таски:
  1. [title] (статус: draft)
  2. [title] (статус: draft)
  ...

Всего: [N] тасок
```

## Фаза 4: Выбор таски для проработки

**Вопрос 3 — Какую таску берём первой?**
«Какую таску прорабатываем первой?»

При выборе:
1. Установить `active_task_id` в выбранную таску
2. Установить таске `status: in_progress`
3. Показать подтверждение:
```
Выбрана таска: [title]
Переходим в pm-copilot-task для проработки ПРД
```
4. Передать контекст в pm-copilot-task:
```
## Контекст из pm-copilot-epic

PM работает с эпиком "[epic.title]" цели "[goal.text]":
- Выбранная таска: [title]
- Другие таски эпика: [список]
- Царь-метрика: [tsar_metric]
- Путь: main (основной)
```

## Цикл: возврат после завершения таски

Когда PRD утверждён (active_prd.status = approved) и PM возвращается в epic:

1. Отметить текущую таску `status: done`, `prd_id: [id]`
2. Показать статус эпика:
```
Эпик: [epic.title]
  [done] [title] — ПРД утверждён
  [draft] [title] — draft
  [draft] [title] — draft

Что дальше?
[1] Следующая таска → pm-copilot-task
[2] Перерыв — сохранить и вернуться позже
```

3. При выборе [1] — установить следующую таску как `active_task_id`, `status: in_progress` → перейти в pm-copilot-task
4. При выборе [2] — сохранить ProductState, показать `Этап: epic (приостановлен)`

## Завершение эпика

Когда все таски эпика имеют `status: done`:
```
Эпик "[epic.title]" завершён! Все [N] тасок проработаны.

Следующий эпик: [next_epic.title] (статус: planned)
Начать? (да / пока нет)
```

При подтверждении → установить следующий эпик как active, начать с Фазы 2.

Когда все эпики цели завершены:
```
Цель "[goal.text]" — все эпики проработаны!
Переходим к comms и запуску
```

## Reflection Checkpoints

Перед Фазой 3 → 4: «Все ли таски учтены? Нет ли пропущенных юзер стори?»

## Phase Limits

При `phase_turns > 3` без сохранения тасок — Anti-Overthinking Guard.

## Чтение ProductState

Этот скилл читает:
- `goals[]` — активная цель с эпиками
- `goals[].active_epic_id` — текущий эпик
- `goals[].epics[].tasks[]` — таски эпика
- `metrics.tsar_metric` — для привязки тасок к царь-метрике
- `product_memory.learned_patterns[]` — для рекомендаций при генерации

## Запись ProductState

Этот скилл пишет:
- `goals[].active_epic_id` — при выборе эпика
- `goals[].epics[].tasks[]` — при сборе тасок
- `goals[].epics[].active_task_id` — при выборе таски
- `goals[].epics[].tasks[].status` — при изменении статуса
- `stage` — обновляет на `epic` при входе, на `task` при передаче
- `path` — устанавливает `main`
- `history[]` — добавляет запись при переходе
- `last_context` — обновляет skill=epic, phase, timestamp
- `updated` — обновляет timestamp

## Переключение

- Если для таски нужна гипотеза → предложи: «Эта таска требует валидации. Запустить короткий путь (/гипотеза)?»
- Если нужна помощь с идеями → вызови generation-инструмент (без смены stage)
