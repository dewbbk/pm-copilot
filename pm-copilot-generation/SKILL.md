# Генерация и идеация

> **Generation — это инструмент, не stage.** Вызывается из любого stage через команду `/идея` или фасадом. После завершения — возвращается в вызвавший stage. Не меняет `stage` в ProductState.
> **Принцип**: Не одна «лучшая» идея, а набор альтернатив. PM выбирает — мы помогаем расширить пространство вариантов перед сужением.

## Два режима работы

Режим определяется по входу:

| Режим | Когда | Вход | Выход | Куда дальше |
|-------|-------|------|-------|-------------|
| **Генерация гипотез** | PM описывает проблему / инсайт / упавшую метрику, но не знает, что проверить | Проблема, метрика, наблюдение | 3-5 проверяемых гипотез | → `pm-copilot-hypothesis` (Путь 2) |
| **Генерация инициатив** | PM описывает подцель / objective, но не знает, какими фичами достичь | Подцель, ключевой результат | 3-7 фич-кандидатов | → Приоритизация → `pm-copilot-task` (Path 3) |

## Приём контекста при переключении

Если активирован после переключения с другого пути — фасад передаст секцию «Контекст из предыдущего пути». Используй её:
- Не задавай вопросы, на которые PM уже ответил в предыдущем пути
- Переформулируй собранные ответы в формат текущего sub-skill
- Начни с первого вопроса, который НЕ покрыт имеющимися ответами
- Покажи PM: «Ты уже говорил, что [суть]. Учту — иду дальше."

### Специальный случай: Контекст из Learning Loop

Если активирован после перехода learning → generation — фасад передаст секцию «Контекст из Learning Loop» с уроками из предыдущих циклов. Используй её:
- **Не предлагай то, что уже не сработало**: Проверь `what_failed` из Learning Cards и исключи эти подходы из генерации
- **Усиливай то, что сработало**: Проверь `what_worked` и предложи вариации успешных подходов
- **Учитывай выявленные паттерны**: Если `learned_patterns` с confidence >= 0.5 — включи их как ограничения/рекомендации при генерации
- Покажи PM: «Из прошлого опыта: [key_learning]. Учту при генерации — не буду предлагать [what_failed], усилю [what_worked]."

Пример:
```
## Контекст из Learning Loop

Уроки из последнего цикла:
- Что сработало: Упрощение флоу даёт +15-25% конверсии
- Что НЕ сработало: Push-уведомления не помогают с удержанием
- Ключевой урок: Упрощение клиентского пути эффективнее механик вовлечения

Выявленные паттерны (confidence >= 0.5):
- Упрощение флоу даёт +15-25% конверсии без роста NPL
```

## Фаза 0: Pre-check из Insight Buffer и Learning Loop

> **v4.16**: Перед началом генерации — проверить Insight Buffer на накопленные инсайты и Product Memory на прошлый опыт.

### Шаг 0a: Insight Buffer Check (v4.16)

Если `insights[]` в ProductState содержит инсайты со `status: new` или `prioritized`:

1. **Загрузить `insights[]`**: Показать PM топ-3 инсайта по `impact_score` (если не приоритизированы — показать все)
2. **Спросить PM**: «У вас есть накопленные инсайты: [список]. Какие учесть при генерации? (все / выборочно / пропустить)»
3. **Если PM выбирает «все»**: Автоматически использовать инсайты как ограничения и направления при генерации в Фазе 2 — генерировать идеи, которые адресуют самые влияющие инсайты
4. **Если PM выбирает «выборочно»**: Спросить, какие именно инсайты учесть
5. **Если PM выбирает «пропустить»**: Начать генерацию без учёта инсайтов (иногда нужно свежее мышление)

Пример:
```
## Контекст из Insight Buffer

Накопленные инсайты:
- «40% клиентов бросают заявку на шаге ввода данных» (влияние: критичное, источник: CustDev)
- «Utilization rate упал в сегменте 18-25» (влияние: прямое, источник: аналитика)
- «Т-Банк запустил мгновенную выдачу» (влияние: косвенное, источник: конкурент)

Используй эти инсайты при генерации: предлагай идеи, которые адресуют самые влияющие инсайты.
```

### Шаг 0b: Learning Loop Check (v4.15)

Если активирован из learning → generation (Learning Loop):

1. **Загрузить `product_memory.learning_cards[]`**: Показать PM топ-3 урока из последней карточки
2. **Загрузить `product_memory.learned_patterns[]`**: Проверить паттерны с confidence >= 0.5
3. **Спросить PM**: «Из прошлого опыта: [key_learning]. Какие уроки учесть при генерации? (все / выборочно / пропустить)»
4. **Если PM выбирает «все»**: Автоматически исключить подходы из `what_failed` и усилить подходы из `what_worked` при генерации в Фазе 2
5. **Если PM выбирает «выборочно»**: Спросить, какие именно уроки учесть
6. **Если PM выбирает «пропустить»**: Начать генерацию без учёта прошлого опыта (иногда нужно свежее мышление)

Если активирован НЕ из learning (обычный путь) — пропустить Фазу 0, перейти сразу к Фазе 1.

> 💾 **Чекпойнт**: Фаза 0 завершена (если применима). Обнови ProductState — если PM выбрал уроки для учёта, зафиксируй это.

## Фаза 1: Отправная точка

### Определение режима

Задай первый вопрос, который одновременно определяет режим:

**Вопрос 1 — Отправная точка:**
«С чем ты пришёл — опиши ситуацию, которую хочешь разрешить. Это может быть проблема, упавшая метрика, инсайт из исследований, подцель, которую нужно достичь, или просто «надо что-то делать с X».»

По ответу определи режим:

| Если PM говорит про | Режим | Пример |
|---------------------|-------|--------|
| Проблему, боль, жалобу, «метрика просела» | Генерация гипотез | «Конверсия в выдачу упала на 12% — не знаем почему» |
| Инсайт, наблюдение, данные | Генерация гипотез | «40% клиентов бросают заявку на шаге ввода данных» |
| Конкурента, «а у них есть» | Генерация гипотез | «Т-Банк показывает предодобренные — может нам тоже?» |
| Подцель, objective, «надо достичь X» | Генерация инициатив | «Нужно увеличить retention D30 с 35% до 50%» |
| Направление, «надо развивать X» | Генерация инициатив | «Надо развивать мобильный онбординг» |
| Неясно | Уточни: «Ты хочешь понять, что происходит (→ генерация гипотез) или знаешь, куда идём, но не знаешь как (→ генерация инициатив)?» | |

Покажи PM: «Понял, ты [описал проблему / назвал подцель]. Я сгенерирую [гипотезы / инициативы] — несколько вариантов, из которых выберем лучшие.»

### Уточнение контекста

**Вопрос 2 — Границы:**
«Какие у нас ограничения и контекст? Что из этого верно?»

| Аспект | Что спрашиваем | Зачем |
|--------|---------------|-------|
| Срок | «К какому сроку нужен результат?» | Отсекает долгосрочные инициативы |
| Ресурсы | «Сколько команды можем выделить?» | Отсекает ресурсоёмкие варианты |
| Домен | «В каких доменах работаем?» (если не из профиля) | Фокусирует идеи |
| Уже пробовали | «Что уже пробовали и не сработало?» | Не генерируем отброшенное |
| Запреты | «Что точно НЕ делаем?» | Явные ограничения |

> **Банковский пример — уточнение контекста**:
>
> PM: «Конверсия в выдачу кредиток упала на 12%»
>
> Уточнение:
> - Срок: «Нужно понять причину за 2 недели, чтобы принять решение до конца квартала»
> - Ресурсы: «Есть 1 аналитик на полставки и 2 разработчика на 1 спринт»
> - Домен: «Кредитные карты + Ориджинейшн» (из профиля)
> - Уже пробовали: «Запускали push-напоминания — не помогло»
> - Запреты: «Не трогаем скоринг-модель — это отдельный проект»

**Вопрос 3 — Данные (для генерации гипотез):**
«Какие данные у нас уже есть по этой проблеме? Метрики, аналитика, отзывы, интервью — что угодно.»

Категории данных:
- **Количественные**: метрики, funnel, когортный анализ, сегментация
- **Качественные**: интервью, отзывы, жалобы, CustDev
- **Рыночные**: конкуренты, тренды, бенчмарки
- **Внутренние**: инциденты, изменения продукта, сезонность, кампании

> **Если данных нет** — это тоже входной сигнал: «Данных пока нет. Первая гипотеза, которую стоит проверить: собрать данные. Без них мы будем генерировать вслепую.» Предложи L0-исследование (кабинетное) как первую гипотезу.

**Вопрос 3b — Ключевые результаты (для генерации инициатив):**
«Какой конкретно результат нужен? Если подцель — то какой ключевой результат (KR) мы пытаемся достичь?»

Формат: «[Метрика] с [текущее] до [целевое] к [срок]»

> **Банковский пример**:
> - «Retention D30 с 35% до 50% к Q3 2026»
> - «Средний остаток на сберегательном счёте с 45 000 ₽ до 65 000 ₽ за 6 месяцев»
> - «Конверсия заявка→выдача с 5% до 7% к концу года»

> 💾 **Чекпойнт**: Фаза 1 завершена. Обнови файл сессии — добавь все собранные ответы. Обнови «Текущую фазу» на «Фаза 2: Генерация».

## Фаза 2: Генерация

> **Ключевой принцип**: Генерация отделена от оценки. На этом этапе НЕ оцениваем идеи — только создаём. Критика позже.

Примени 6 техник генерации последовательно. Для каждой техники:
1. Объясни технику в 1-2 предложениях
2. Сгенерируй 2-4 варианта на основе контекста PM
3. Запиши все варианты — не фильтруй

### Техника 1: Инверсия

«Представь обратную ситуацию. Что нужно сделать, чтобы проблема стала ХУЖЕ? А потом инвертируй — что из этого подсказывает решение?»

| Шаг | Вопрос | Банковский пример |
|-----|--------|-------------------|
| Что ухудшит? | «Что бы ты сделал, чтобы конверсия упала ЕЩЁ сильнее?» | «Сделать заявку ещё длиннее, убрать предодобренные, добавить обязательные поля, скрыть кнопку "Далее"» |
| Инверсия | Противоположность каждого пункта | «Сократить заявку, показать предодобренные, убрать лишние поля, сделать CTA заметнее» |
| Гипотезы/инициативы | Каждая инверсия → кандидат | Гипотеза: «Сокращение шагов заявки с 7 до 4 увеличит конверсию на X%» |

> **Почему работает**: Мозг легче находит ошибки и ухудшения, чем улучшения. Это когнитивный трюк — через негативное мышление приходим к позитивным идеям.

### Техника 2: Аналогия

«Где эта проблема уже решена — в смежном домене, у конкурентов, в другой индустрии?»

| Источник аналогии | Вопрос | Банковский пример |
|-------------------|--------|-------------------|
| Конкуренты | «Как эту проблему решают Т-Банк, Альфа, Сбер?» | «Т-Банк: предодобренные на главном экране + 3-шаговая заявка. Альфа: кросс-селл в мобильном после транзакции» |
| Смежный домен | «А в других доменах банка?» | «В ипотеке — предодобренные работают. Почему в кредитках нет? Что мешает перенести механику?» |
| Другая индустрия | «А за пределами банка?» | «E-commerce: Amazon 1-click, Ozon Express — одномоментная покупка без корзины. Что если кредитка — «одним кликом»?» |
| Внутренние кейсы | «Что уже работает в твоём продукте, что можно расширить?» | «Push-напоминания для дебетовых работают (open rate 25%). Можно ли адаптировать для кредиток?» |

> **Важно**: Аналогия — не рецепт. «У конкурентов так» ≠ «Нам так надо». Каждая аналогия становится гипотезой, которую нужно валидировать в нашем контексте.

### Техника 3: Декомпозиция проблемы

«Разбей проблему на части. Какой вклад каждая часть вносит в общую проблему? Решаем по частям.»

**Для генерации гипотез** — декомпозируем funnel / процесс:

```
Общая проблема: Конверсия упала на 12%
     ↓
Шаг 1: Лендинг → заявка: -3%  ← Причина: баннер не видят?
Шаг 2: Заявка → скоринг: -5%  ← Причина: сложная форма?
Шаг 3: Скоринг → оферта: -2%  ← Причина: долгий скоринг?
Шаг 4: Оферта → подписание: -1% ← Причина: условия не устраивают?
Шаг 5: Подписание → выдача: -1% ← Причина: доставка?
```

Для каждого шага с наибольшим вкладом → отдельная гипотеза.

**Для генерации инициатив** — декомпозируем целевую метрику:

```
Цель: Retention D30 с 35% до 50% (+15 п.п.)
     ↓
Новые клиенты: +5 п.п. ← Лучший онбординг, early value
Месяц 1-2: +4 п.п. ← Привыкание, первые транзакции
Месяц 2-3: +3 п.п. ← Engagement-механики, push
Спящие: +3 п.п. ← Реактивация, win-back
```

Для каждого слота → отдельная инициатива.

> **Банковский пример — декомпозиция funnel**:
>
> Проблема: конверсия заявка→выдача кредиток упала с 5.0% до 4.4%
>
> Декомпозиция:
> - Шаг «Ввод данных»: completion rate 60% → гипотеза: автозаполнение из Госуслуг поднимет до 75%
> - Шаг «Скоринг»: время решения 4 часа → гипотеза: мгновенный скоринг поднимет конверсию (клиенты не уходят к конкурентам пока ждут)
> - Шаг «Оферта»: take rate 40% → гипотеза: персонализированный лимит + кешбэк-стартер поднимет take rate до 55%
> - Шаг «Выдача»: доставка 5 дней → гипотеза: цифровая выдача за 1 день поднимет конверсию на 3 п.п.

### Техника 4: Крайние ограничения

«Что если бы у нас был только 1 день? Только 0 бюджета? Или наоборот — бесконечный бюджет? Какие идеи приходят в голову?»

| Ограничение | Вопрос | Что даёт |
|-------------|--------|----------|
| **1 день** | «Какую гипотезу/инициативу можно проверить/запустить за 1 день?» | Самые дешёвые и быстрые варианты — часто самые умные |
| **0 бюджета** | «Что можно сделать, не тратя ни копейки?» | Использование существующих механик, каналов, данных |
| **Бесконечный бюджет** | «Если бы ресурсы не были проблемой — что бы ты сделал?» | Амбициозные идеи, которые можно упростить до реалистичных |
| **Только аналитика** | «Что можно сделать, используя только данные, которые уже есть?» | L0-гипотезы без разработки |

> **Банковский пример**:
>
> Проблема: конверсия кредиток упала
>
> - **1 день**: Посмотреть funnel по сегментам — может, падение только в одном сегменте? → Гипотеза: «Падение сконцентрировано в сегменте X»
> - **0 бюджета**: Изменить текст push-уведомления о предодобренном → Гипотеза: «Текущий текст push не мотивирует»
> - **Бесконечный бюджет**: Полный предодобренный конвейер с мгновенной выдачей → Инициатива, которую можно упростить: предодобренный флоу для 10% трафика
> - **Только аналитика**: Когортный анализ — когда именно начали отваливаться? → Гипотеза: «Падение началось после изменения X»

### Техника 5: Customer Journey-дыры

«Пройди по клиентскому пути и найди «дыры» — моменты, где клиент теряется, ждёт, не понимает, злится. Каждая дыра — кандидат на гипотезу или инициативу.»

Для банковских продуктов типичный клиентский путь:

```
Привлечение → Первый контакт → Заявка → Скоринг → Оферта → Подписание → Активация →
Использование → Повторные транзакции → Кросс-селл → Удержание → (Отток → Возврат)
```

Для каждого этапа задай вопросы:

| Этап | Вопросы-зонды |
|------|---------------|
| Привлечение | «Откуда клиент приходит? А откуда мог бы? Какой канал недоиспользуем?» |
| Первый контакт | «Клиент видит продукт — что он думает? Понятно ли, что это? Зачем ему это?» |
| Заявка | «Сколько шагов? Где отваливаются? Сколько времени занимает? Какой шаг самый болезненный?» |
| Скоринг | «Сколько ждёт? Что делает пока ждёт? Что думает? Уходит ли к конкурентам?» |
| Оферта | «Понятны ли условия? Соответствуют ли ожиданиям? Есть ли выбор?» |
| Подписание | «Как подписывает? Удобно ли? Не пугает ли юридический язык?» |
| Активация | «Сделал ли первое действие? Понимает ли, что делать дальше? Как быстро получает ценность?» |
| Использование | «Как часто использует? Что мешает использовать чаще? Что помогает?» |
| Кросс-селл | «Знает ли о других продуктах? Как ему предлагают? Уместно ли предложение?» |
| Удержание | «Почему остаётся? Почему уходит? Что делает конкурент, что мы не делаем?» |

> **Банковский пример — дыры в journey кредиток**:
>
> | Дыра | Признак | Гипотеза / Инициатива |
> |------|---------|----------------------|
> | Клиент не видит предодобренное | CTR баннера < 1% | Гипотеза: «Баннер ниже fold — клиенты не скроллят» |
> | Клиент бросает на шаге «Работа и доход» | Dropout 35% на этом шаге | Гипотеза: «Автозаполнение из 2-НДФЛ через Госуслуги сократит dropout» |
> | Клиент ждёт решение 4 часа | 20% уходят к конкурентам | Инициатива: мгновенный скоринг |
> | Клиент не активирует карту | Activation rate 45% | Инициатива: бонус за первую покупку в течение 7 дней |
> | Клиент не понимает кешбэк | Only 15% выбирают категории | Инициатива: авто-выбор категорий на основе трат |

### Техника 6: Метрический драйвер

«Какие драйверные метрики влияют на царь-метрику? Для каждого драйвера — что может его сдвинуть?»

Используй `pm-copilot/references/metrics.md` для загрузки иерархии метрик продукта PM.

```
Царь-метрика: Profit per card
     ↓ Драйверы
┌────────────┬────────────┬────────────┬────────────┐
│ Utilization │ Roll rate  │ Avg balance│ Activation │
│  30%        │  20%       │  25%       │  25%       │
└────────────┴────────────┴────────────┴────────────┘
     ↓            ↓            ↓            ↓
 Что может     Что может     Что может     Что может
 поднять?      снизить?      вырастить?    поднять?
```

Для каждого драйвера — генерируй инициативы, которые на него влияют:

| Драйвер | Инициативы |
|---------|-----------|
| Utilization | Персональные лимиты, промо-периоды, кешбэк на большие покупки |
| Roll rate | Напоминания о платеже, автопогашение, реструктуризация до просрочки |
| Avg balance | Перевод зарплаты на карту, seamless-пополнение, round-up |
| Activation | Бонус за первую покупку, onboarding-тур, push «не забыли про карту?» |

> **Банковский пример — метрический драйвер для аквизиции**:
>
> Царь-метрика: CAC
> - Драйвер: CPA по каналам → инициатива: оптимизация воронки лендинга (снижаем CPA за счёт конверсии, не за счёт снижения ставки)
> - Драйвер: Конверсия лендинг→заявка → инициатива: предодобренный лендинг вместо универсального
> - Драйвер: Конверсия заявка→выдача → инициатива: сокращение заявки с 7 до 4 шагов
> - Драйвер: Retention D30 → инициатива: onboarding-флоу с 3 ключевыми действиями в первые 7 дней

## Фаза 4: Приоритизация и хендоф

### Для генерации гипотез — быстрый приоритет

Для оставшихся гипотез предложи матрицу:

| Гипотеза | Ожидаемая ценность | Стоимость проверки | Обучающий потенциал | Приоритет |
|----------|-------------------|-------------------|--------------------|-----------|
| [суть 1] | Высокая / Средняя / Низкая | L0(1-3 дня) / L1(1-2 нед) / L2(2-4 нед) / L3(4-8 нед) | Высокий / Средний / Низкий | 1 / 2 / 3 |

**Правило приоритизации:**
1. Сначала L0 — дешёвые кабинетные исследования. Если данные уже есть — проверяем за 1-3 дня.
2. Потом L1 — если L0 не опроверг.
3. L2/L3 — только если L0+L1 дают сигнал.

> **Банковский пример — приоритизация гипотез**:
>
> | # | Гипотеза | Ож. ценность | Стоимость | Обучение | Приоритет |
> |---|----------|-------------|-----------|----------|-----------|
> | 1 | «Падение конверсии сконцентрировано в сегменте X» | Средняя | L0 (1 день) | Высокое | 1 |
> | 2 | «Автозаполнение из Госуслуг поднимет completion rate» | Высокая | L2 (4 нед) | Среднее | 3 |
> | 3 | «Баннер предодобренных ниже fold — клиенты не видят» | Средняя | L0 (2 дня) | Среднее | 2 |
> | 4 | «Мгновенный скоринг устраняет отток» | Высокая | L3 (8 нед) | Высокое | 4 |

Покажи PM: «Рекомендую начать с гипотезы [N] — она проверяется за [срок] и даст понимание, куда копать дальше. Переключаемся на валидацию?»

### Для генерации инициатив — RICE-lite

Для оставшихся инициатив предложи быструю оценку:

| Инициатива | Reach (охват) | Impact (влияние) | Confidence (уверенность) | Effort (затраты) | RICE score |
|-----------|---------------|-----------------|-------------------------|-------------------|------------|
| [суть 1] | [кол-во людей] | 3=массивное, 2=среднее, 1=малое | 3=высокая, 2=средняя, 1=низкая | [чел-месяцы] | (R×I×C)/E |
| [суть 2] | ... | ... | ... | ... | ... |

> **Упрощение**: Полный RICE требует точных цифр, которых часто нет. Используй RICE-lite — относительные оценки вместо абсолютных. PM ранжирует, а не считает.

Покажи PM топ-3 инициативы: «Три самых перспективных инициативы: [список с RICE score]. Какую берём в проработку первой?»

## Антипаттерны

### Антипаттерн 1: «Генерация в подтверждение»

PM приходит с уже готовым решением в голове и использует генерацию как «подтверждение» — ищет только те идеи, которые совпадают с его задумкой.

**Как распознать**: PM отвергает все идеи, кроме одной, которая совпадает с тем, что он уже хотел. Фразы: «Ну это не то», «Мне нужно именно про X».

**Как исправить**: «Вижу, что тебя тянет к варианту [X]. Это нормально — но давай честно: ты пришёл за идеями или за подтверждением? Если за подтверждением — давай сразу пойдём в валидацию гипотезы [X]. Если за идеями — дай мне шанс показать альтернативы, прежде чем отбросишь.»

### Антипаттерн 2: «Паралич выбора»

Сгенерировано 15+ вариантов, и PM не может выбрать. Чем больше вариантов — тем сложнее решение.

**Как распознать**: PM говорит «надо подумать», «все интересные», не может назвать топ-3.

**Как исправить**: Принудительное сужение — «Давай выберем не топ-3, а ровно ОДНУ гипотезу/инициативу, которую проверим первой. Остальные не пропадут — они в файле сессии. Какую проверим прямо сейчас?»

### Антипаттерн 3: «Бесконечная генерация»

PM хочет «ещё вариантов» — каждый раз, когда видит список, просит сгенерировать ещё. Это форма прокрастинации: генерация чувствуется как прогресс, но это иллюзия.

**Как распознать**: Второй или третий запрос «давай ещё» без выбора из предыдущих.

**Как исправить**: «Мы уже сгенерировали [N] вариантов. Больше вариантов не значит лучше решение — значит сложнее выбор. Давай перейдём к фильтрации и выберем из того, что есть. Если ни один не подходит — это сигнал, что мы неправильно сформулировали отправную точку. Вернёмся к Фазе 1?»

### Антипаттерн 4: «Идея без связи с метрикой»

PM генерирует идеи, которые «круты», но не влияют на царь-метрику. «Давайте добавим тёмную тему» — да, приятно, но на Profit per card не влияет.

**Как распознать**: При формулировке идеи PM не может построить цепочку: идея → прокси → драйвер → царь-метрика.

**Как исправить**: Для каждой идеи спроси: «Как это влияет на [царь-метрика]? Если не можешь построить цепочку — идея может быть классной, но она не для текущей цели. Запишем в бэклог и вернёмся, когда будет релевантно.»

### Антипаттерн 5: «Копирование без контекста»

PM берёт аналогию (обычно у конкурента) и хочет скопировать один-в-один без адаптации к своему контексту. «Т-Банк так делает — давайте так же».

**Как распознать**: Фразы «у конкурентов это работает», «давайте как у X».

**Как исправить**: «У [конкурента] это работает в их контексте: [другая аудитория / другая архитектура / другой стадия продукта]. Давай сформулируем, ПОЧЕМУ это работает у них, и проверим, действует ли тот же механизм у нас. Это будет наша гипотеза — не «сделать как у них», а «проверить, работает ли тот же драйвер в нашем контексте».»

Этот скилл читает из ProductState:

## Запись ProductState

Этот скилл пишет в ProductState:
- `hypotheses[]` — добавляет сгенерированные гипотезы (с текстом, статусом `draft`, начальной уверенностью)
- `updated` — обновляет временную метку
- **Важно: `stage` НЕ меняется** — generation это инструмент, вызвавший stage остаётся активным
- `last_context` — обновляет skill=generation, phase, timestamp
