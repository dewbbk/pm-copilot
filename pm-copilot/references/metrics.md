# Metrics Framework — Фреймворк метрик

> Single source of truth. Тесты и скиллы ссылаются на этот файл.
> Версия: v4.13 (Sprint 21-pre: Domain consolidation — pm-copilot-domain → references)

## Область применения

Фасад и все workflow-скиллы используют этот фреймворк для работы с метриками. Команда `метрики` загружает этот файл. Все скиллы, которым нужен контекст царь-метрик и ганардов, ссылаются сюда.

## Принцип царь-метрик

**Царь-метрика** — единственная метрика, которая определяет успех продукта в текущем периоде. Всё остальное — прокси, драйверы или операционные метрики, которые важны только постольку, поскольку они двигают царь-метрику.

**Правило**: Одна царь-метрика на продукт за период. Нельзя иметь 5 царь-метрик — это значит, что нет ни одной.

## Иерархия метрик

```
┌─────────────────────────────────┐
│        ЦАРЬ-МЕТРИКА             │  Одна. Определяет успех.
│   (North Star Metric)           │  Пример: LTV клиента
├─────────────────────────────────┤
│     ДРАЙВЕРНЫЕ МЕТРИКИ          │  3-5 метрик, которые
│   (Input Metrics)               │  двигают царь-метрику
│   Пример: Retention, Cross-sell │  и на которые можно влиять
│   rate, Transaction frequency   │
├─────────────────────────────────┤
│     ПРОКСИ-МЕТРИКИ              │  Операционные метрики,
│   (Proxy Metrics)               │  которые коррелируют
│   Пример: DAU, NPS,            │  с драйверными, но сами
│   Conversion rate               │  царь-метрику не двигают
├─────────────────────────────────┤
│     ДИАГНОСТИЧЕСКИЕ МЕТРИКИ     │  Метрики для понимания
│   (Diagnostic Metrics)         │  «почему», не для принятия
│   Пример: Time on screen,      │  решений
│   Bounce rate, Error rate       │
└─────────────────────────────────┘
```

### Царь-метрика
- Одна на продукт/инициативу
- Отражает главный бизнес-результат
- Примеры: Profit per card, LTV, Conversion rate, MAU

### Драйверные метрики
- 3-5 метрик, которые двигают царь-метрику
- Связь: «если драйверная растёт, царь-метрика тоже растёт»
- Примеры: Utilization rate, Roll rate, Activation rate

### Прокси-метрики
- Операционные метрики, которые косвенно влияют на драйверные
- Используются для краткосрочного мониторинга
- Примеры: CTR кнопки, время заполнения формы, bounce rate

## Типичные царь-метрики по банковским продуктам

### Ритейл

| Продукт | Царь-метрика | Драйверные метрики |
|---------|-------------|---------------------|
| Дебетовые карты | LTV клиента | Retention, Transaction frequency, Cross-sell rate |
| Кредитные карты | Profit per card | Utilization rate, Roll rate, Average balance |
| Кредиты наличными | NPV кредитного портфеля | Approval rate, NPL, Average ticket |
| Ипотека | Объём выдачи при целевом NPL | Approval rate, Time-to-decision, Conversion из заявки |
| Вклады | Объём привлечённых средств | Conversion открытия, Average balance, Retention |

### МСБ

| Продукт | Царь-метрика | Драйверные метрики |
|---------|-------------|---------------------|
| РКО | Количество активных клиентов | Activation rate, Transaction volume, Churn rate |
| Эквайринг | Turnover per merchant | Activation, Average ticket, Churn rate |
| Кредиты бизнесу | NPV портфеля | Approval rate, NPL, Cross-sell |

### Диджитал

| Продукт | Царь-метрика | Драйверные метрики |
|---------|-------------|---------------------|
| Мобильное приложение | MAU активных пользователей | DAU/MAU ratio, Session depth, Feature adoption |
| Интернет-банк | Доля транзакций в диджитал | Adoption rate, Task completion rate, Contact rate |

## Метрики по фазам клиентского пути

| Фаза | Метрика | Что измеряет |
|------|---------|-------------|
| Привлечение | CPA, Conversion в установку | Стоимость и эффективность привлечения |
| Онбординг | Time-to-value, Completion rate | Скорость и успешность подключения |
| Активация | First transaction rate, Days to first action | Переход от регистрации к использованию |
| Транзакции | Frequency, Volume, ARPU | Активность использования |
| Кросс-селл | Penetration rate, Products per client | Глубина проникновения продуктов |
| Удержание | Retention rate, Churn rate | Долгосрочная привязка |
| Возврат | Win-back rate, Reactivation rate | Возвращение ушедших |

## Ганарды (Guardrail Metrics)

Ганарды = метрики, которые НЕ должны ухудшиться при реализации фичи.

| Тип | Описание | Пример |
|-----|----------|--------|
| Финансовый | Прямые потери | NPL <= 4.5%, Fraud rate <= 0.08% |
| Клиентский | Удовлетворённость | NPS >= 38, App store rating >= 4.5 |
| Операционный | Процессные лимиты | Time-to-approve <= 5 мин |
| Регуляторный | Compliance | Compliance rate >= 99.9% |

**Правило**: Если задача улучшает целевую метрику, но ухудшает ганард — это не улучшение, это перетягивание каната. Нужно искать решение, которое не ломает ганарды.

## Как связать задачу с царь-метрикой

Для каждой задачи PM должен заполнить цепочку:

```
Задача → Прокси-метрика → Драйверная метрика → Царь-метрика
```

**Пример:**
```
Задача: Упростить заявку на кредитную карту
Прокси: Конверсия шага «подтверждение» с 45% до 60%
Драйвер: Conversion из заявки в выдачу +5 п.п.
Царь: Profit per card +8%
```

Если PM не может построить цепочку до царь-метрики — это сигнал, что:
1. Задача не влияет на царь-метрику (может, не стоит делать)
2. PM не понимает, как влияет (нужно копать глубже)
3. Царь-метрика не определена для продукта (нужно сначала определить)

## Расчёт эффекта: шаблон

```markdown
## Расчёт эффекта задачи

### Царь-метрика
[Название]: текущее [значение] → целевое [значение] = разница [значение]

### Цепочка влияния
Задача: [описание]
Прокси-метрика: [название] с [текущее] до [целевое]
Драйверная метрика: [название] с [текущее] до [целевое]
Царь-метрика: [название] с [текущее] до [целевое]

### Расчёт
[Прокси] разница[значение] x [коэффициент влияния на драйвер] = разница[драйвер]
[Драйвер] разница[значение] x [коэффициент влияния на царь] = разница[царь]

### Ганарды
[Метрика-ганард 1]: текущее [значение], порог ухудшения [не более чем на X%]
[Метрика-ганард 2]: текущее [значение], порог ухудшения [не более чем на X%]

### Уверенность
- Влияние на прокси: [Высокая/Средняя/Низкая]
- Влияние прокси на драйвер: [Высокая/Средняя/Низкая]
- Влияние драйвера на царь: [Высокая/Средняя/Низкая]
- Общая уверенность: [Высокая/Средняя/Низкая]
```

## Когда нет метрик

Если PM не может назвать метрики:

1. **Нет данных** → Сформируй запрос Data-аналитику
2. **Нет понимания, что мерить** → Помоги определить царь-метрику продукта и спустись вниз
3. **Нет базовой линии** → Предложи сначала замерить текущее состояние (1-2 недели) и потом ставить цель
4. **Нет целевого значения** → Предложи ориентиры:
   - Исторический рост продукта (как росло в прошлом)
   - Бенчмарк рынка (как у конкурентов)
   - Минимально значимый эффект (какой рост уже стоит усилий)

## Правила для скиллов

1. Любой эффект привязывай к царь-метрике
2. Если PM не знает царь-метрику — помоги сформулировать
3. Прокси-метрики — только как шаг к царь-метрике (не самоцель)
4. Ганарды — всегда спрашивай «что НЕ должно ухудшиться»
5. Если фича может ухудшить ганард — предупреждай явно
