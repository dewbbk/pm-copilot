# Metrics Framework — Фреймворк метрик

> Версия: v6.2 (Sprint 30: сжатие)

## Принцип царь-метрик

**Царь-метрика** — единственная метрика успеха продукта в текущем периоде. Одна. Нельзя иметь 5 царь-метрик.

**Иерархия**: Царь-метрика → Драйверные (3-5, двигают царь-метрику) → Прокси (косвенно влияют) → Диагностические (для понимания «почему»).

**Банковские примеры:**
- Царь: CAC на активного клиента, Количество SPEND NTB клиентов
- Драйверные: CR Digital funnel, Delivery Rate, Activation rate
- Прокси: CTR кнопки, время заполнения формы, bounce rate

## Цепочка влияния

Для каждой задачи: `Задача → Прокси → Драйверная → Царь-метрика`

Если цепочку построить нельзя — задача либо не влияет на царь-метрику, либо царь-метрика не определена.

## Ганарды

Метрики, которые НЕ должны ухудшиться. Для банка типичные: NPL, Fraud rate, NPS, Compliance rate, Contact rate.

Всегда спрашивай: «Что НЕ должно ухудшиться?»

## Правила для скиллов

1. Любой эффект привязывай к царь-метрике
2. Если PM не знает царь-метрику — помоги сформулировать
3. Прокси-метрики — только как шаг к царь-метрике
4. Если фича может ухудшить ганард — предупреждай явно
5. При отсутствии метрик — предложи сначала замерить (1-2 недели baseline)
