## Что такое CAP

CAP (Context Annotation Platform) — платформа, на которой собрана информация обо всех процессах в компании. Это единая точка правды о том, как устроена компания «под капотом».

**Для PM CAP важен тем, что:**
- Каждый процесс банка задокументирован и связан с другими
- Можно проследить цепочку от клиентского действия до бэк-офисной обработки
- Видны зависимости между системами и процессами
- Понятны владельцы процессов и стейкхолдеры

**Как использовать CAP в работе PM:**
1. Перед запуском задачи — найди свой процесс в CAP
2. Посмотри смежные процессы (upstream/downstream)
3. Определи, какие системы затронуты
4. Найди владельцев смежных процессов — их нужно подключить

## Структура банковского домена

### Уровень 1: Направления бизнеса

| Направление | Описание | Типичные продукты |
|-------------|----------|-------------------|
| Ритейл | Физлица, массовый рынок | Карты, вклады, кредиты наличными, ипотека |
| МСБ | Малый и средний бизнес | РКО, эквайринг, кредиты бизнесу |
| КБ | Корпоративный бизнес | Кредитные линии, гарантии, Cash Management |
| Private | VIP / HNWI | Индивидуальное обслуживание, инвестиции |
| Диджитал | Цифровые каналы | Мобильное приложение, Интернет-банк, OpenAPI |

### Уровень 2: Процессы по типу

#### Клиентский путь (Front-office)

```
Привлечение → Онбординг → Активация → Транзакционная активность → Кросс-селл → Удержание → Возврат (если ушёл)
```

Каждый этап — это отдельный процесс в CAP с владельцем, метриками и системами.

#### Кредитный конвейер

```
Заявка → Скоринг → Кредитный комитет → Оферта → Подписание → Выдача → Обслуживание → Погашение → Работа с просрочкой
```

#### Операционные процессы (Back-office)

```
Открытие счёта → Настройка продуктов → Проведение операций → Реконсиляция → Закрытие
```

#### Процессы комплаенса и риска

```
KYC/AML → Мониторинг транзакций → Отчётность ЦБ → Внутренний аудит → Управление рисками
```

### Уровень 3: Системный ландшафт

| Слой | Типичные системы | Роль для PM |
|------|------------------|-------------|
| Каналы | Мобильное приложение, ИБ, API-шлюз | Где клиент видит изменения |
| Процессинг | Процессинг карт, АБС, процессинг кредитов | Где исполняется бизнес-логика |
| Платформы | ESB, API-платформа, Data Platform | Как системы общаются между собой |
| Данные | DWH, DMP, аналитические витрины | Где считаются метрики |
| Фронт-офис | CRM, система диалогов, колл-центр | Где сотрудники работают с клиентом |

## Типовые зависимости между процессами

При проработке задачи PM обязан проверить:

### Upstream-зависимости (от кого зависим)
- Данные поступают из [система/процесс]
- Событие инициируется [процесс]
- Решение зависит от [скоринг/комитет/система]

### Downstream-зависимости (кто зависит от нас)
- Результат потребляется [система/процесс]
- Событие триггерит [процесс]
- Изменение влияет на [интеграция/API]

### Cross-зависимости (смежные процессы)
- Параллельно идёт [процесс] — нужно синхронизировать
- Тот же клиент проходит через [процесс] — нужен единый опыт
- Те же данные используются в [процесс] — консистентность

## Регуляторный контекст

Банк — одна из самых зарегулированных индустрий. PM обязан понимать:

| Регулятор | Что контролирует | Как влияет на продукт |
|-----------|------------------|----------------------|
| ЦБ РФ | Лицензирование, нормативы, отчётность | Ограничения на продукты, сроки, тарифы |
| Росфинмониторинг | AML/Финтерроризм | KYC-процедуры, мониторинг транзакций |
| ФАС | Конкуренция, реклама | Ограничения на маркетинг, тарифы |
| Роспотребнадзор | Права потребителей | Условия договоров, отказы |
| 152-ФЗ | Персональные данные | Сбор, хранение, обработка данных |

**Правило для PM**: Любая задача, затрагивающая клиентские данные, условия продуктов или клиентскую коммуникацию, потенциально имеет регуляторный аспект. Если сомневаешься — привлеки комплаенс.

## Как искать информацию в CAP

Когда PM упоминает процесс, продукт или систему:

1. **По процессу**: Какой бизнес-процесс? Как он называется в CAP? Кто владелец?
2. **По продукту**: Какой банковский продукт? В какой системе он обслуживается? Какие смежные продукты?
3. **По системе**: Какая система? Какие процессы через неё проходят? Кто ответственная команда?
4. **По данным**: Какие данные? Откуда берутся? Кто владелец? Где хранятся?

Если CAP доступен для поиска — используй его напрямую. Если нет — формируй запрос для архитектора/аналитика:
```markdown
## Запрос в архитектуру

**Контекст**: [какая задача, зачем нужно]
**Что ищем**: [процесс/система/данные]
**Что нужно узнать**:
- Связанные процессы в CAP
- Затронутые системы
- Стейкхолдеры
- Регуляторные ограничения
```

## Словарь банковских терминов

| Термин | Значение | Контекст для PM |
|--------|----------|-----------------|
| АБС | Автоматизированная банковская система | Ядро банка, где ведутся счета и проводятся операции |
| Скоринг | Оценка кредитоспособности | Определяет, дать или не дать кредит |
| KYC | Know Your Customer | Процедура идентификации клиента |
| AML | Anti-Money Laundering | Противодействие отмыванию денег |
| NPL | Non-Performing Loan | Доля просроченных кредитов — ключевая метрика риска |
| NIM | Net Interest Margin | Чистая процентная маржа — ключевая метрика доходности |
| LTV | Lifetime Value | Пожизненная ценность клиента |
| CAC | Customer Acquisition Cost | Стоимость привлечения клиента |
| RKO | Расчётно-кассовое обслуживание | Базовый продукт для бизнеса |
| ESB | Enterprise Service Bus | Шина интеграции между системами |
| API-шлюз | Точка входа для внешних вызовов | OpenAPI, партнёрские интеграции |
| Реконсиляция | Сверка данных между системами | Критична для консистентности |
| Провайд | Проводка, банковская операция | Атомарная единица движения денег |
| ПТС | Продуктовый технологический сервис | Микросервис, реализующий продуктовую логику |

## Царь-метрики по доменам

| Домен | Код | Царь-метрика | Ключевые драйверы |
|-------|-----|-------------|-------------------|
| Кредитные карты | 1 | Profit per card | Utilization rate, Roll rate, Average balance, Activation rate |
| Ориджинейшн/Онбординг | 17 | Conversion rate | Time-to-approve, Drop-off rate, KYC completion |
| Мобильное приложение | 21 | MAU/DAU | Session length, Feature penetration, Push opt-in |
| CVM | 18 | LTV | Share of wallet, Retention, Penetration |
| Энгейджмент | 19 | Engagement score | NBA hit rate, Feature adoption, Cross-sell rate |

## Ганарды по доменам

| Домен | Типичные ганарды |
|-------|-----------------|
| Кредитные карты | NPL <= 4.5%, Fraud rate <= 0.08%, NPS >= 38 |
| Ориджинейшн | Dropout rate <= X, Compliance score >= Y |
| Мобильное приложение | Crash rate <= 0.1%, App store rating >= 4.5 |

## Правила для скиллов

1. При упоминании процесса — найти в CAP и предложить смежные
2. Метрики — от царь-метрик вниз (прокси — только как шаг к царь-метрике)
3. Ганарды — всегда спрашивать «что НЕ должно ухудшиться»
4. Регуляторика — предупреждать о рисках, но не давать юридических консультаций
5. Если PM не может построить цепочку до царь-метрики — это сигнал, что задача может быть не нужна или царь-метрика не определена
