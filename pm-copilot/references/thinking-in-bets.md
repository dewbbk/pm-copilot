# Thinking-in-Bets — Вероятностный фреймворк

> Single source of truth. Тесты и скиллы ссылаются на этот файл.
> Версия: v5.1 (Fix Sprint: обновлена версия reference)

## Область применения

Суб-скилл `pm-copilot-thinking-in-bets` использует этот фреймворк для вероятностного анализа решений. Фасад подключает TIB:

**Автоматически (v4.09)**:
- confidence < 0.7 при фиксации Decision → быстрый режим
- Решение затрагивает царь-метрику → полный режим
- Past decisions с actual ≠ expected → полный режим

**Вручную**:
- Явная просьба PM ("вероятностный анализ", "премортем", "полный анализ")

## Два режима TIB

### Быстрый режим

Для мелких решений (confidence < 0.7, царь-метрика не затронута):

```
🎲 Быстрый вероятностный анализ

Confidence: [значение]

Сценарии:
  ✅ Ожидаемый: [описание] — вероятность [X]%
  ⚠️ Худший: [описание] — вероятность [Y]%

Что делать если худший сценарий: [митигация]

Провести полный анализ? (да/нет)
```

Содержит: confidence + 2 сценария + митигация. Не содержит: premortem, EV, групповую валидацию.

### Полный режим

Для стратегических решений (царь-метрика, прошлые сюрпризы, ручной запрос «полный анализ»):

1. 4 сценария (лучший / ожидаемый / худший / чёрный лебедь)
2. Премортем
3. Ожидаемая ценность (EV)
4. «Почему я могу ошибаться?»
5. Групповая валидация
6. Цепочка решений (при наличии related_decisions)

## 4 сценария

| Сценарий | Описание | Вероятность |
|----------|----------|-------------|
| Лучший | Всё идёт по плану, все допущения верны | 10-20% |
| Ожидаемый | Базовый сценарий, реализуется чаще всего | 50-60% |
| Худший | Ключевые допущения не верны | 15-25% |
| Чёрный лебедь | Непредсказуемое событие с катастрофическими последствиями | 1-5% |

Сумма вероятностей должна быть ~100%.

## Премортем

Формат: "Задача запущена и провалилась. Что пошло не так?"

PM должен назвать >=3 конкретных причины неудачи. Общие формулировки ("не хватило ресурсов") не принимаются — нужны конкретные ( "data-аналитик Марина не смогла подготовить отчёт из-за приоритизации другого проекта").

## Expected Value

```
EV = P(best) * V(best) + P(expected) * V(expected) + P(worst) * V(worst) + P(black_swan) * V(black_swan)
```

Где V = ценность исхода (может быть отрицательной).

## Триггеры автоподключения

| # | Условие | Предложение | Режим TIB | Приоритет |
|---|---------|-------------|-----------|-----------|
| 1 | confidence < 0.7 | «Уровень уверенности низкий. Провести вероятностный анализ?» | Быстрый | Низший |
| 2 | Решение затрагивает царь-метрику | «Это решение влияет на царь-метрику. Проверить сценарии?» | Полный | Высший |
| 3 | Past decisions с actual ≠ expected | «Похожее решение в прошлый раз дало неожиданный результат. Пересмотреть?» | Полный | Средний |

Приоритет: 2 > 3 > 1. Если несколько триггеров — показать одно предложение по приоритету.

## Использование цепочек решений

TIB читает `related_decisions` из Decision Log:

- **Цепочка confidence**: Если все решения в цепочке имеют confidence < 0.7 — усиливаем предупреждение
- **Прошлые сюрпризы**: Ищем decisions с `actual_outcome ≠ expected_outcome` в цепочке
- **Царь-метрика**: Проверяем, какие решения в цепочке уже анализировали царь-метрику

Формат в артефакте «Карта ставок»:

```markdown
## Цепочка решений

Решение основано на:
1. [decision-id] hypothesis: «[краткое содержание]» (confidence: 0.7)
2. [decision-id] goal: «[краткое содержание]» (confidence: 0.65)
3. → Текущее: [decision-id] task: «[краткое содержание]» (confidence: 0.55)

⚠️ Уровень уверенности в цепочке падает. Рекомендуется полный TIB.
```

## Использование Product Memory

TIB проверяет `product_memory.learned_patterns` при анализе:
- Если есть паттерн, связанный с текущим решением — включает его в сценарии
- Например: «Раньше кэшбэк-кампании давали +5% utilization, но рос roll rate» → учитываем в худшем сценарии

## Связь с Decision Log

Каждый вероятностный анализ связан с записью в Decision Log через `decision_id`. При заполнении `actual_outcome` — сравниваем с ожидаемым сценарием.
