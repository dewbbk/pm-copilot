# Decision Log — Схема записи решения

> Single source of truth. Тесты (pm-copilot-tests) и скиллы ссылаются на этот файл.
> Версия: v4.08 (Sprint 17: Decision Linking — связи между решениями)

## Формат Decision

```yaml
Decision:
  id: string                          # уникальный ID (автогенерация: decision-YYYY-MM-DD-HHMM)
  timestamp: datetime                 # когда принято
  stage: enum                         # на каком этапе принято (insight | goal | hypothesis | task | comms | launch | post-launch | learning)
  decision: string                    # что решили (краткая формулировка)
  rationale: string                   # почему (обоснование)
  assumptions: []                     # какие допущения легли в основу
  expected_outcome: string            # что ожидали от решения
  actual_outcome: string | null       # что вышло в реальности (заполняется в post-launch)
  confidence: 0-1                     # уверенность на момент решения (0=полная неуверенность, 1=абсолютная уверенность)
  alternatives: []                    # отвергнутые альтернативы (что ещё рассматривали)
  status: enum                        # open | reviewed | superseded (открытое | проверенное | заменённое другим решением)
  related_decisions: []               # ID связанных решений — предыдущее решение, на котором основано текущее (v4.08)
  supersedes: string | null           # ID решения, которое текущее отменяет/заменяет (null = не заменяет) (v4.08)
```

## Жизненный цикл Decision

1. **Создание**: Workflow-скилл фиксирует ключевое решение → создаёт Decision с `status: open`, `actual_outcome: null`
2. **Накопление**: Решения хранятся в Decision Log. Команда `решения` показывает все.
3. **Ревизия**: Команда `ревизия` показывает только `open`-решения (без actual_outcome).
4. **Заполнение actual_outcome**: При post-launch — Copilot заполняет actual_outcome, меняет `status: reviewed`
5. **Замена**: Если новое решение отменяет предыдущее → `status: superseded`, ссылка на новый Decision
6. **Связывание**: При создании нового Decision — автоматически связать с предыдущим решением на том же ProductState через `related_decisions` (v4.08)

## Интеграция с ProductState

Decision Log хранится в `ProductState.decisions[]` как список ссылок (id). Полные записи решений хранятся отдельно. При команде `решения` — Copilot раскрывает все записи.

## Команды фасада

### `решения` — показать все решения по продукту

Выводит таблицу всех решений в ProductState:

```
📋 Decision Log: [название продукта]

| # | Решение | Stage | Уверенность | Статус | Дата |
|---|---------|-------|-------------|--------|------|
| 1 | Запустить кэшбэк на рестораны | hypothesis | 0.7 | open | 2026-05-11 |
| 2 | Декомпозировать на 3 эпика | goal | 0.8 | open | 2026-05-11 |

Всего: 2 решения | Открытых: 2 | Проверенных: 0
```

При наличии `actual_outcome` — добавляет колонку с результатом.

### `ревизия` — показать открытые решения

Показывает только решения без `actual_outcome` (статус `open`):

```
🔍 Открытые решения (ревизия): [название продукта]

| # | Решение | Допущения | Ожидаемый исход | Confidence |
|---|---------|-----------|-----------------|------------|
| 1 | Запустить кэшбэк на рестораны |_utilization вырастет на 5%_ | +5% utilization | 0.7 |

Всего открытых: 1
Команда `ревизия` помогает PM отслеживать, какие решения ещё не проверены.
```

## Точки записи решений (триггеры)

| Скилл | Триггер | Что записывается | Интеграция |
|-------|---------|-----------------|------------|
| **hypothesis** | PM валидирует/инвалидирует гипотезу | decision = результат, rationale = почему, confidence = из вероятностной оценки | ✅ v4.04 |
| **goal** | PM выбирает ключевые результаты и эпики | decision = выбранные KR/эпики, rationale = почему эти, alternatives = отвергнутые | ✅ v4.04 |
| **task** | PM фиксирует ценность/эффект задачи | decision = ценность + эффект, rationale = почему это важно, assumptions = допущения о влиянии на метрику | ✅ v4.04 |
| **post-launch** | PM оценивает результаты | actual_outcome = что вышло, заполняется у соответствующего Decision; новый Decision при решении scale/pivot/kill | ✅ v4.04 |

## Правила записи

1. **Не каждое сообщение — решение**. Записываем только ключевые точки: validate/invalidate, выбор KR, фиксация ценности, оценка результата.
2. **Copilot предлагает записать** — не записывает молча. Формат: «Записать это решение? (да/нет)». PM подтверждает.
3. **confidence заполняется автоматически** из вероятностной оценки (если была) или спрашивается у PM.
4. **alternatives** — Copilot подсказывает отвергнутые варианты из контекста диалога.
5. **actual_outcome** — заполняется только при post-launch, не раньше.

## Decision Linking — Связи между решениями

> **Ядро v4.08**: Decisions — больше не плоский список. Каждое решение связано с предыдущим на том же ProductState, образуя направленный граф от гипотезы до post-launch.

### Логика автосвязывания

При создании нового Decision Copilot автоматически связывает его с предыдущим решением на том же ProductState:

| Скилл | related_decisions | Объяснение |
|-------|-------------------|------------|
| **goal** | `[]` (пустой) | Начало цепочки — цель первична, нет предыдущего решения на этом ProductState |
| **hypothesis** | `[ID от goal Decision]` | Гипотеза обслуживает цель — проверяет неопределённые эпики |
| **task** | `[ID от goal Decision]` | Задача вытекает из декомпозиции цели |
| **post-launch** | `[ID от task Decision]` | Оценка результатов задачи |

### Поле `supersedes`

Если новое решение отменяет/заменяет предыдущее (например, PM пересмотрел ценность задачи):
- Старый Decision: `status: superseded`
- Новый Decision: `supersedes: [ID старого Decision]`

Это не то же самое, что `related_decisions` — `supersedes` означает замену, а `related_decisions` — логическую последовательность.

### Команда `цепочка [decision-id]`

Показывает полную цепочку связанных решений от начального до текущего:

```
🔗 Цепочка решений: [decision-id]

1️⃣ goal: «Выбраны 4 KR и 9 эпиков для цели LTV +20%...» (confidence: 0.7) ✅ reviewed
    ↓
2️⃣ hypothesis: «Гипотеза валидирована: кэшбэк на рестораны...» (confidence: 0.65) ✅ reviewed
    ↓
3️⃣ task: «Фиксация ценности: упрощение заявки...» (confidence: 0.55) 🟡 open
    ↓
4️⃣ post-launch: «Масштабируем кэшбэк на 100% трафика» (confidence: 0.7) ✅ reviewed

Цепочка: 4 решения | Открытых: 1 | Проверенных: 3
```

Если Decision имеет `supersedes` — показывать ответвление:

```
2️⃣ goal: «Выбраны KR...» → ЗАМЕНЕНО на ↓
2️⃣' goal: «Пересмотрены KR: фокус на retention...» (supersedes: [2️⃣])
```

### Правила Decision Linking

1. **Автосвязывание при создании.** При создании нового Decision — Copilot ищет последний Decision на предыдущем stage в том же ProductState и добавляет его ID в `related_decisions`. Если не найден — `related_decisions = []`.
2. **PM может убрать связь.** Если автосвязывание некорректно — PM может командой «убрать связь с [decision-id]» очистить related_decisions.
3. **`цепочка` показывает полный путь.** Команда `цепочка` идёт по related_decisions рекурсивно, пока не дойдёт до Decision с `related_decisions = []` (начало цепочки).
4. **`supersedes` — для замен, не для последовательности.** Когда PM пересматривает решение — старое помечается `superseded`, новое получает `supersedes`. Не путать с `related_decisions`.
5. **TIB использует цепочки.** При автоподключении (Sprint 18) TIB сможет анализировать всю цепочку решений, а не только текущее.

## Связь со скиллами

- **goal**: выбор KR и эпиков → записать, `related_decisions = []` (начало цепочки — цель первична) ✅ v4.12
- **hypothesis**: решение validate/invalidate → записать, `related_decisions = [ID от goal Decision]` (гипотеза обслуживает цель) ✅ v4.12
- **task**: фиксация ценности/эффекта → записать, `related_decisions = [ID от goal Decision]` ✅ v4.08
- **post-launch**: при оценке результатов → заполнить actual_outcome + новый Decision при scale/pivot/kill, `related_decisions = [ID от task Decision]` ✅ v4.08
- **thinking-in-bets**: вероятностная оценка → источник confidence и alternatives. Будет использовать цепочки решений (Sprint 18)

## Связь с PM Memory (стиль решений)

> **Ядро v4.06**: Decision Log — источник данных для автоанализа стиля решений. Каждая запись в Decision Log обогащает профиль PM.

Decision Log используется для определения стиля решений PM:

| Поле Decision | Что анализируется для стиля | Параметр стиля |
|---------------|---------------------------|----------------|
| `confidence` | Доля решений с confidence < 0.5 vs ≥ 0.7 | `risk_tolerance` |
| `rationale` | Наличие только подтверждающих аргументов → confirmation_bias; отсутствие альтернатив → anchoring | `typical_biases` |
| `alternatives` | Пустые alternatives → anchoring; короткие alternatives → surface depth | `preferred_depth`, `typical_biases` |
| `assumptions` | Количество и качество допущений → surface vs deep | `preferred_depth` |
| `actual_outcome` | Паттерн «итерировать» при негативном результате → sunk_cost | `typical_biases` |

**Правило обновления**: Каждые 5+ решений в Decision Log → Copilot предлагает PM обновить стиль решений (см. фасад, секция PM Memory).
