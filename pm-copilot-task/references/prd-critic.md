# PRD Critic — Аудит качества ПРД

> **Когда использовать**: После генерации ПРД (Фаза 5), перед оценкой соответствия царь-метрикам (Фаза 6). Это тактический аудит — можно ли отдавать ПРД в разработку прямо сейчас или есть дыры.

## Зачем нужен PRD Critic

Фаза 6 оценивает ПРД **стратегически** — двигает ли царь-метрику, стоит ли делать. Но даже стратегически верный ПРД может быть **тактически неготовым** — с пробелами, которые заставят разработку возвращаться с вопросами. PRD Critic — это чеклист качества ПРД как документа, не как решения.

Каждый найденный пробел ДО передачи в разработку экономит дни переделок.

## 6 измерений оценки

### 1. Ясность проблемы

**Проверь**: Определена ли проблема, которую решаем? Понятно ли, почему это проблема, а не просто «хотелка»?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| ПРД начинается с решения | «Нам нужна кнопка X» — а зачем? Какая проблема? |
| Нет подтверждения проблемы | Нет данных, отзывов, метрик, подтверждающих что проблема существует |
| Проблема описана для банка, не для клиента | «Нам нужно увеличить кросс-селл» — а что клиенту мешает? |
| Проблема слишком широкая | «Плохой UX» — где конкретно, для кого, как измеряется? |

**Зелёный сигнал**: Проблема сформулирована через опыт клиента, подтверждена данными, конкретна.

### 2. Границы (Scope)

**Проверь**: Чётко ли определено, что входит и НЕ входит? Где заканчивается эта задача?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| Нет секции «Не входит» | Scope будет ползти — каждый добавит «ну, ещё вот это» |
| Размытые формулировки | «Улучшить процесс» — какие именно шаги? |
| Скрытый scope | «Ну и заодно подтянем дизайн» — это отдельная задача? |
| Неясно, что с существующим флоу | Старый путь остаётся? Удаляется? Для кого? |

**Зелёный сигнал**: Scope/Non-scope прописаны, границы по продуктам/сегментам/каналам определены.

### 3. Критерии приёмки

**Проверь**: Есть ли измеримые «сделано, когда» условия? Может ли разработчик и PM одинаково понять, готова ли фича?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| Качественные, не количественные | «Улучшить UX» — как измерим? «Снизить время заявки с 8 до 3 минут» — вот это критерий |
| Нет критериев для ганардов | Целевые метрики есть, а пороги для ганардов не определены |
| Нет критерия «не готовы» | При каком результате откатываем? |
| Критерии только функциональные | А производительность? Безопасность? Доступность? |

**Зелёный сигнал**: Каждый критерий измерим, имеет пороговое значение, понятно кто и как проверяет.

### 4. Edge cases

**Проверь**: Рассмотрены ли краевые и нестандартные ситуации? Что происходит за «happy path»?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| Только happy path | А если клиент прервал заявку? Если нет сети? Если данные не пришли? |
| Нет обработки ошибок | Что видит пользователь при ошибке API? При таймауте? |
| Не учтены особые сегменты | VIP-клиенты, юридические лица, клиенты с ограничениями — как для них? |
| Не рассмотрен обратный переход | Что если клиент захочет вернуться к старому флоу? |

**Зелёный сигнал**: Основные edge cases описаны, есть план обработки ошибок, учтены особые сегменты.

### 5. Метрики

**Проверь**: Определены ли метрики успеха достаточно полно? Можно ли по ним принять решение?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| Нет царь-метрики | К какому результату продукта привязано? |
| Нет базовой линии | «Увеличим конверсию на 20%» — с какого уровня? С 10% или с 50%? |
| Нет ганардов | Целевая метрика может вырасти за счёт риска/качества |
| Нет методики замера | Из какой системы? Какой запрос? Какой период? Кто считает? |
| Метрики только внутренние | А клиентские? NPS, satisfaction, task completion? |

**Зелёный сигнал**: Цепочка влияния построена (Задача → Прокси → Драйвер → Царь), ганарды определены, методика замера ясна.

### 6. Неоднозначности и пробелы

**Проверь**: Есть ли места, которые разные люди прочитают по-разному? Чего не хватает?

| Сигнал проблемы | Что искать |
|-----------------|-----------|
| Двоякие формулировки | «Упростить» — для кого? На сколько? Что именно? |
| Не указаны стейкхолдеры | Кто ещё должен согласовать? Кто владелец смежного процесса? |
| Нет зависимостей | От кого/от чего зависит реализация? Что если зависимость не будет готова? |
| Нет допущений | На что рассчитываем, но не проверили? |
| Много [УТОЧНИТЬ] | Если больше 3 — ПРД не готов |

**Зелёный сигнал**: Минимум неоднозначностей, допущения явно обозначены, зависимости указаны.

## Формат результата аудита

После проверки ПРД по 6 измерениям сформируй:

```markdown
# Аудит ПРД: [Название]

## Сильные стороны
[Что в ПРД сделано хорошо — по каждому измерению где пройдено]

## Пробелы и риски
| Измерение | Пробел | Риск если не исправить |
|-----------|--------|----------------------|
| [измерение] | [чего не хватает] | [что будет если отдать так] |

## Уточняющие вопросы
[Вопросы, на которые нужно ответить прежде чем передавать в разработку]

## Рекомендации по доработке
[Конкретные предложения — что добавить/изменить в ПРД]

## Вердикт
**Готов к разработке** / **Требует доработки** / **Требует существенной переработки**
```

## Правила проведения аудита

1. **Аудит ≠ Переписывание** — не переписывай ПРД за PM, укажи пробелы и дай рекомендации
2. **Конкретно, не абстрактно** — не «нужно лучше описать проблему», а «в секции 1.1 нет данных, подтверждающих масштаб проблемы — добавь статистику дроп-оффа»
3. **Приоритизируй пробелы** — не все пробелы равны. Отсутствие edge cases — это одно, отсутствие царь-метрики — критично
4. **Проверяй на «прочтёт ли разработчик»** — разработчик не знает контекста сессии. Всё ли понятно из документа?
5. **Считай [УТОЧНИТЬ]** — если больше 3 пометок — ПРД не готов, нужно вернуться к PM

## Связь с Фазой 6

PRD Critic — это **под-этап** Фазы 6, который проводится ПЕРЕД стратегической оценкой:

```
Фаза 6: Оценка ПРД
  ├── Шаг 1: PRD Critic (тактический аудит качества документа)
  ├── Шаг 2: Оценка по царь-метрикам (стратегическая оценка)
  ├── Шаг 3: Финальная валидация + красные флаги
  └── Шаг 4: Групповая валидация
```

Если PRD Critic выдал «Требует существенной переработки» — нет смысла переходить к стратегической оценке. Сначала приведи документ в порядок.
